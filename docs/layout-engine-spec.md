# layout-engine-spec.md — 規則式佈局引擎 Implementation Brief

## 1. Purpose

本文件是**本輪唯一的實作依據**，定義規則式佈局引擎要做什麼、產出什麼行為。這是單輪實作文件，完成並驗收後即成為歷史參考，不做長期維護。

## 2. Authority and References

本輪的文件優先層級：

- **本文件（layout-engine-spec.md）** 定義本輪的目標行為。若本文件與其他文件衝突，以本文件為準。
- **layout-direction.md** 定義設計規則——構圖、間距、配色比例、驗收標準。本文件不重複貼其內容，只引用段落編號。
- **asset-interface.md** 定義花型資產的技術約束——載入路徑、幾何結構、朝向、正規化保證、換色介面、效能上限。佈局引擎產生的任何花朵擺放與換色操作，都必須遵守此文件列出的「現有程式碼已強制」項目。
- **Codebase** 定義現有實作事實。程式碼裡實際存在的東西，才是現況。
- **ADR** 定義策略決策。D4 策略一 = 規則式佈局引擎（本輪執行的即是此策略）。

## 3. Goal

**本輪 = ADR §6 第 4 步：策略一，規則式佈局引擎。**

產出三種佈局原型——新月弧、雙群花圈、滿版圓頂。每種佈局可以一鍵生成合法的 flower JSON，並在場景中渲染到蛋糕上。

**本輪不做：** LLM 佈局生成（策略二）、花朵品質升級、程序化花生成、光照材質升級、手機觸控優化、佈局參數的使用者 UI。

## 4. Scope

### In scope

- layout-direction.md §1 三種佈局原型的定義與辨識特徵。
- layout-direction.md §2A / §2B / §2C 各佈局的構圖規則：焦點花位置、擴散方向、大小遞減、花型角色分配、覆蓋面積、花流到側面的範圍。
- layout-direction.md §3 花與花之間的空間規則：最小間距、bounding box 重疊容忍、朝向規則。
- layout-direction.md §4 可放置區域：蛋糕頂面、蛋糕側面、底緣與底盤交界、底盤表面（僅 fivepetal 或花瓣，數量上限 3–5）、蛋糕底面不放花。
- layout-direction.md §5A / §5B / §5C 與「跨佈局通用」的驗收標準。
- layout-direction.md §7 配色與視覺比例：色彩面積分佈、空間分區、同色系深淺分配、深淺穿插規則、大花小花視覺層次、奶白色角色。佈局引擎需讀取 Screen 1 已配好的花色資料並依此規則分配花朵的空間位置，不重新設計配色本身。

### Out of scope

- LLM 或其他生成式 AI 驅動的佈局生成（ADR 策略二）。
- 花朵模型品質升級、程序化花生成（新花型建模）。
- 光照與材質系統升級。
- 手機觸控操作優化。
- 佈局參數的使用者可調 UI（所有可調數值寫死在程式碼常數）。
- layout-direction.md §6 列出的所有項目：花的生長/排列動畫、佈局之間的混合或漸變、蛋糕抹面顏色選擇、葉片、底盤散落花瓣裝飾、圖四側面散點構圖（高蛋糕多層佈局）。

## 5. Required Impact Analysis Before Coding

**分析完成並經使用者確認前，不要寫或修改任何程式碼。**

產出以下四份清單：

**List A — 依賴掃描：** 所有讀取或寫入花朵放置資料（flower JSON 中的位置、旋轉、所屬花盤/蛋糕狀態）、拖曳吸附邏輯（DragManager 或同等模組）、花盤庫存（FlowerTray 或同等模組）、Screen 2 進入點與渲染迴圈的程式碼位置。每一項列出檔案路徑、函式/元件名、用途。

**List B — 需修改的現有檔案：** 為了讓佈局引擎能以程式方式呼叫現有的放置與吸附邏輯（而非僅回應使用者手動拖曳），哪些現有檔案需要新增呼叫介面或調整。列出檔案路徑與修改內容摘要。

**List C — 新建檔案：** 佈局演算法模組（新月弧、雙群花圈、滿版圓頂）、硬約束檢查器（驗證 §9 中「可程式檢查」的項目）、佈局選擇 UI 元件。每個列出檔案路徑與內容摘要。

**List D — Assumptions / Open Questions：** 分析過程中發現的不確定點（例如：花盤庫存是否固定數量、flower JSON 是否已有可用來標記「花所屬佈局群組」的欄位）。

產出四份清單後停下來，等待使用者確認。確認前不開始寫程式碼。

## 6. Target Behavior

使用者在 Screen 2 看到一個佈局選擇 UI。UI 最小化，不遮擋蛋糕視野，提供三個選項對應三種佈局原型。

選擇一種佈局後，花朵自動從花盤移動並吸附到蛋糕上，不需要使用者逐朵拖曳。排列結果套用 Screen 1 已配好的花色。

排列完成後，使用者仍可對個別花朵手動調整：拖曳移動、換色、移除。手動調整不受佈局引擎鎖定，行為與現有拖曳吸附系統一致。

三種佈局各自的生成行為：

- **新月弧：** 依 layout-direction.md §2A。焦點花位置、擴散方向（頂面到側面的對角帶）、大小遞減比例、花型角色分配（peony/rose 起頭、rose 中段、fivepetal/hydrangea 收尾）、覆蓋面積 30%–40%（可調）、花流延伸到側面高度 0%–50%（可調）。
- **雙群花圈：** 依 layout-direction.md §2B。兩個花群焦點位置與角度差 150°–210°（可調）、各花群擴散半徑、連接段花朵、覆蓋面積 40%–55%（可調）、中央留白直徑 35%–50%（可調）、花群外緣越過頂面到側面高度 0%–20%（可調）。
- **滿版圓頂：** 依 layout-direction.md §2C。焦點花位置偏心約 1/4 半徑、不規則填充擴散、覆蓋面積 80%–95%（可調）、dome 高度輪廓、邊緣花垂落側面高度 10%–30%（可調）。

三種佈局都須遵守 layout-direction.md §3（間距與朝向）與 §4（可放置區域）。

## 7. Data Model Changes

佈局引擎的 input 和 output 都是現有的 flower JSON 格式。**不改動現有資料結構。**

若 impact analysis（List D）發現需要一個欄位來標記「此花由哪個佈局生成」以利除錯或未來手動調整時的行為區分，列為開放問題，於確認階段與使用者討論是否新增，不在本輪自行決定新增欄位。

## 8. Implementation Requirements

### Must preserve

- 所有現有互動：拖曳、吸附、選色、截圖、reset。
- Screen 1 配色流程與其輸出格式。
- asset-interface.md 列出的所有「現有程式碼已強制」約束——GLB 路徑規則、Mesh/material 結構、朝向假設、正規化原點保證、單一顏色輸入介面。

### Must build new

- 佈局演算法模組：三種佈局原型各自的花朵位置/旋轉/尺寸計算邏輯。
- 佈局選擇 UI：最小化，不遮擋蛋糕。
- 硬約束檢查器：驗證產出結果符合 §9 中標記「可程式檢查」的驗收標準（重疊比例、角度差、覆蓋面積、焦點花數量為奇數等）。

### Must avoid

- 不碰花的模型載入邏輯（GLB 載入、normalizeFlower、材質清除流程）。
- 不碰配色系統（Screen 1 的漸層生成、色票邏輯）。
- 不碰 Screen 1 本身的畫面與流程。
- 不做 layout-direction.md §6 列出的任何項目。

## 9. Acceptance Criteria

### 5A — 新月弧

- 花流覆蓋蛋糕頂面面積不超過 40%，頂面至少 55% 為連續裸露抹面區域。**可程式檢查**（計算花的 bounding box 在頂面的投影面積佔比）。
- 花流帶狀形狀有明確厚端與薄端，厚端花群寬度至少是薄端的 2 倍。**可程式檢查**（比較帶狀兩端花群寬度比）。
- 花流不是直線，沿蛋糕邊緣有弧度；花從頂面到側面自然過渡，不在邊緣突然斷裂。**手動目測**。

### 5B — 雙群花圈

- 兩個花群焦點花的角度差在 120°–210° 之間。**可程式檢查**。
- 頂面中央留白區域直徑不小於蛋糕直徑的 30%。**可程式檢查**（中央區域內無花的 bounding box）。
- 兩花群之間有可見的串聯花朵，不是兩個孤立花堆；整體看得出「環」的暗示。**手動目測**。

### 5C — 滿版圓頂

- 花覆蓋頂面面積至少 80%。**可程式檢查**（花的 bounding box 在頂面的投影覆蓋率）。
- 焦點花最高點距離圓心至少 15% 半徑，不在正中心。**可程式檢查**。
- 俯視看不到大面積抹面；側視花的輪廓呈 dome 形，中間高、邊緣低。**手動目測**。

### 跨佈局通用

- 沒有任何兩朵同種花的 bounding box 重疊超過 30%。**可程式檢查**。
- 沒有任何兩朵同種花的 Y 軸旋轉角度差小於 25°。**可程式檢查**。
- 焦點花數量為奇數（1 或 3）。**可程式檢查**。

### 本輪額外自動驗證

- `tsc --noEmit` 或等效編譯檢查通過，無型別錯誤。**可程式檢查**。
- grep 確認佈局引擎模組未 import 任何 Screen 1 配色元件或花朵模型載入模組（驗證 §8 Must avoid 的邊界未被跨越）。**可程式檢查**。
- 三種佈局個別選取後，Screen 2 渲染無 console error。**手動目測**。

## 11. Non-Goals

- 不做 LLM 或其他生成式方法的佈局生成（策略二，未來另立文件）。
- 不做花朵模型品質提升或程序化花生成。
- 不做光照與材質系統升級。
- 不做手機觸控操作優化。
- 不開放佈局參數給使用者調整的 UI。
- 不做花的生長或排列動畫。
- 不做佈局之間的混合或漸變。
- 不做蛋糕抹面顏色選擇。
- 不做葉片。
- 不做底盤散落花瓣裝飾。
- 不做高蛋糕多層側面散點構圖。
