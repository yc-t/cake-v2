# 韓式裱花蛋糕 3D 設計工具 — 策略決策記錄（Decision Record）

> 文件性質：ADR（Architecture Decision Record）+ 產品策略摘要。
> 用途：任何 AI session 或協作者開工前必讀，避免重新推導已定案的決策。
> 最後更新：2026-07-14 ｜ 狀態：已定案，待 repo 連入後產出 B 方向 spec

---

## 1. 產品脈絡

瀏覽器內 3D 韓式裱花（豆沙花）蛋糕設計工具。使用者可自由旋轉相機、拖曳花朵、單朵選取、換色；未來支援調整大小與方向。目標裝置：一般筆電與手機瀏覽器，同場景 15–30 朵花。

視覺基準是**韓式裱花作品照**（霧面、擠花紋理、層疊、局部雙色），**不是真花照片**。

既有系統（在另一 Opus 4.8 session 開發，含 spec）：flower JSON 資料結構（position / rotation / scale / type / baseColor）、拖曳、吸附、選色、漸層功能均已可運作。

## 3. 已定案的決策與理由

### D1. 第一測試方向 = B（AI 佈局引擎），不是 A（花朵逼真化）
- **理由**：B 的成敗由已 debug 的既有系統決定（迴路是通的）；A 的成敗由拿不到的資產資料決定（per-petal 幾何缺失，shader 繞不過）。
- B 同時弱化 A 的痛點：整顆蛋糕遠看是構圖在說話，單朵花瓣紋理不再致命。
- Fable 級 one-shot 案例（霍格華茲城堡等）證明的是「程式架構與場景」的 one-shot 能力，**不是**「通過領域專家檢視的資產保真度」——此區分是 A/B 排序的核心依據。

### D2. 場景氛圍與 B 同步做，但美術方向 = 「風格化一致」，不是寫實
- **理由**：觀感由一致性決定，不由絕對精度決定。低保真花 + 照片級場景 = 更醜。
- 基準：柔光產品攝影風（柔光、淺景深、bloom、粉彩調色、乾淨背景）——夢幻感恰好是對低保真資產最寬容的風格，柔光天然藏拙。
- **禁止**：HDR 寫實環境貼圖、銳利高對比陰影（會放大幾何缺陷）。

### D3. 在既有 codebase 上改，不從零重寫
- **理由**：重寫只有在舊架構擋住新目標時才正當；此處舊架構（flower JSON + 互動）正是 B 的跑道。
- 行動：把 Opus session 中的 spec 與結論**提煉成 repo 內文件**。知識存 repo，不存聊天視窗。

### D4. B 之下的兩條策略，依序執行，各有停損
- **策略一：規則式佈局引擎**（先做）。3–4 種佈局原型（單焦點放射、新月弧、滿鋪花牆等），參數化構圖規則，輸出既有 flower JSON。零外部依賴、可重現。
  - 停損：每次生成都需大改才能看 → 規則化投報率太低，棄。
- **策略二：LLM 氛圍 → 佈局 JSON**（策略一成功後）。自然語言（「清晨花園感」）→ 合法佈局 JSON。策略一的產出當 few-shot 範例；策略一的硬約束檢查器復用為 **validator**（不重疊、不出界、不飄空），LLM 輸出不合法自動重試。
  - 停損：過 validator 仍構圖平庸、每次需人工重排 → 退回策略一當唯一引擎。

### D5. 花朵品質升級 = 可熱插拔的延後層
- B 的引擎只操作 flower JSON（型號+位置+旋轉+縮放+顏色），引用資產、不關心資產內部 → 未來換高品質花模型，佈局程式一行不改。
- **資產介面契約（B spec 必寫）**：
  1. 錨點：花底中心為原點（Y_min = 0，XZ 中心 ≈ 0）
  2. 朝向：Y 軸朝上，root node rotation = identity
  3. 尺度：以 rose 直徑 ~0.44 為基準單位，各花型目標直徑 0.35–0.55
  4. 換色參數槽：至少內外雙色（colorInner / colorOuter）
- 不寫此契約 = 未來每換一朵花都要改佈局程式。

**現有四模型實測（2026-07-16，@gltf-transform 解析，transform 已展開至世界座標）**：

| 模型 | BBox 尺寸 X × Y × Z | Y_min（花底） | XZ 中心偏移 | Root node transform | 不符項目 |
|---|---|---|---|---|---|
| fivepetal | 1.09 × 0.10 × 1.15 | 0.000 ✅ | (+0.055, 0) ≈ ✅ | identity ✅ | 高度極薄（0.10）；直徑 ~1.1，為 rose 的 2.5× |
| rose | 0.43 × 0.23 × 0.45 | 0.830 ❌ | (+0.007, −0.011) ✅ | rot = [−0.707, 0, 0, 0.707]（−90° X）❌ | 花底離原點 Y=0.83；root rotation 非 identity |
| hydrangea | 0.27 × 0.18 × 0.26 | 0.286 ❌ | (+0.000, +0.010) ✅ | rot = −90° X + scale = 0.018 ❌ | 花底離原點 Y=0.29；root scale = 0.018 非 1 |
| peony | 1.15 × 0.93 × 1.12 | −0.132 ❌ | (+0.099, +0.223) ❌ | 複合四元數 [−0.33, 0.26, 0.29, 0.86] ❌ | 花底低於原點；XZ 明顯偏心；尺寸為 rose 的 2.5×；旋轉任意 |

**結論**：四朵均不符合「花底 Y=0」契約；僅 fivepetal 的 root 為 identity。B 引擎載入時須對各模型注入修正用的 wrapper transform（平移 + 旋轉 + 縮放），或由美術預先整理 GLB，使其符合契約。

### D6. 程序化生成花：降級為「有停損的 spike」，非已驗證方案
- 結構性優勢真實存在（per-petal ID、UV 方向、風格統一、天生適合 instancing 與參數化配色），但輪廓品質未驗證。
- 參考照 + 截圖迭代迴路（AI 可自行渲染截圖比對參考照）把它從「不能承諾」升級為「值得認真 spike」，但不承諾「近距離騙過裱花老師」——最後 10% 是雕塑問題，非參數問題。
- 正式品質的備援路徑：人工 Blender 重做核心花型並烘入 petal 資料（vertex color / 第二 UV）。

## 4. 參考照片收集規格（供程序化 spike 用，可與 B 開發並行收集）

**每一種花型 8–12 張**（非全部共用；新花型照同規格補一組）：

- 單朵特寫 5–6 張：正上方俯視、45° 斜側、近水平側面三角度必備
- 雙色/漸層特寫 2–3 張：能看清單片花瓣內的顏色過渡方向
- 整顆蛋糕 2–3 張：看組裝比例、間距（兼作 B 的佈局參考）

格式：jpg/png、長邊 ≥1000px、對焦清楚、光線均勻。加分：同朵多角度連拍、標註花嘴型號。避免：重濾鏡、過曝反光。用途限內部參考，不進產品展示。

## 5. 效能預算（草案，需依目標機型確認）

- 整場景 ≤ 300–500k tris（30 朵 rose 級 ≈ 45 萬，已臨界）
- draw calls < 100（hydrangea 現況一朵即 254，必須合併 + KTX2 壓貼圖）
- 中階手機 Chrome ≥ 30 FPS，首次載入 < 5 秒
- 互動約束：單朵選取/拖曳/換色 → 不同花不可 merge 成單一 mesh；同款花不同色 → instancing 需 per-instance color attribute，**架構期就要設計**

## 6. 執行順序

1. ✅ 資產實測審查（本文件 §2）
2. ⬜ 連入專案 repo → 驗證實際 flower JSON schema
3. ⬜ 撰寫 B spec：佈局規格（原型+參數+驗收）+ 資產介面契約（D5）+ 硬約束清單
4. ⬜ 策略一：規則式佈局引擎 開發與驗證
5. ⬜（並行）收集參考照片（§4）；30 分鐘搜尋市售裱花模型排除路 2
6. ⬜ 策略二：LLM 佈局生成（策略一成功後）
7. ⬜ 程序化花 spike（照片齊後）：與現有 rose+新材質對照，盲評 3 項（輪廓/色彩分布/豆沙質感）各 1–5 分，任一方案三項 ≥3 過關；spike 迭代 3 輪輪廓仍 ≤2 → 停止，轉 Blender 人工路線
8. ⬜ 花資產熱插拔升級（D5 契約保證零改動）

## 7. 明確不承諾清單

- AI 無人工介入做出「近距離通過裱花老師檢視」的花
- rose/peony 焊死 mesh 上的每瓣雙色條紋
- 30 朵花在手機上的精細花瓣自陰影（以貼圖 AO + 蛋糕面 shadow map 折衷）
- hydrangea 不經重處理直接使用
- 免費模型的商用授權（未查）
